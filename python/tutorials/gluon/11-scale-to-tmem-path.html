关键对比：A/B operand 走 TMA 一步直达 SMEM；而 scale 这一行最长、最绕 —— GMEM → 寄存器 → convert_layout → 再写进 TMEM。这正是 simple kernel 慢的根源，也是后面三步优化要逐段砍掉的对象。

② scale 这条路的四跳，对应源码（simple_mma_scaled_kernel）

1

GMEM → 寄存器

gl.load 用一个 2D BlockedLayout 把 scale 读进寄存器。沿 K 的连续维只有 BLOCK_K//VEC_SIZE 个字节，远小于 128B 缓存行。

不合并 · 抖 L2
2

convert_layout

load 用的寄存器布局 ≠ TMEM 要求的布局，必须 convert_layout 在线程间重排数据。

寄存器 shuffle
3

寄存器 → TMEM

scale_tmem.store(...) 写进张量内存，布局必须是 TensorMemoryScalesLayout。

local store 不优
4

TMEM → tensor core

硬件 tcgen05_mma_scaled 直接从 TMEM 读 scale，和 SMEM 里的 operand 一起算。

# 1) TMEM 里给 scale 开空间，布局固定为 TensorMemoryScalesLayout
scale_layout = TensorMemoryScalesLayout()
a_scale_tmem = allocate_tensor_memory(dtype, [BLOCK_M, BLOCK_K // VEC_SIZE], scale_layout)
b_scale_tmem = allocate_tensor_memory(dtype, [BLOCK_N, BLOCK_K // VEC_SIZE], scale_layout)

# --- 内层 k 循环里 ---
# hop1: GMEM -> 寄存器（BlockedLayout，连续维太短 → 不合并）
a_scale = gl.load(a_scale_ptr + offs_m[:,None]*stride_m + offs_k[None,:]*stride_k)

# hop2: 转成 TMEM 期望的寄存器布局
a_scale = gl.convert_layout(a_scale, a_scale_tmem.get_reg_layout())

# hop3: 寄存器 -> TMEM
a_scale_tmem.store(a_scale)

# hop4: 硬件从 SMEM(operand) + TMEM(scale) 读入，写累加器到 TMEM
tcgen05_mma_scaled(a_smem, b_smem.permute((1,0)), acc_tmem,
                   a_scale_tmem, b_scale_tmem, a_format, b_format, use_acc=use_acc)

③ 为什么 scale 必须进 TMEM、必须用 TensorMemoryScalesLayout

事实原因 / 约束来源
scale 住在 TMEM（不是 SMEM）硬件 tcgen05_mma_scaled 规定 scale operand 从张量内存读取；operand 才从 SMEM 读。
布局必须 TensorMemoryScalesLayout这是硬件能识别的 scale 排布；用别的布局 store 进 TMEM 后 MMA 读不对。
B scale 永远按 [BLOCK_N, BLOCK_K//VEC_SIZE] 传入指令对 B scale 的形状约定（即使 B operand 是转置喂入）。
BLOCK_N ≥ 32TensorMemoryScalesLayout 写入 TMEM 时至少需要 32 行；A scale 有 128 行（因 BLOCK_M=128）不受限，但 B scale 的行数 = BLOCK_N，故 BLOCK_N 不能小于 32。
BLOCK_M = 128使用 tcgen05_mma_scaled 的硬性要求。
累加器也在 TMEM（TensorMemoryLayout, fp32）Blackwell 累加器常驻 TMEM —— 这点后面让它比 Hopper 能多 overlap 一截。
所以上一张图里 BLOCK_M=128 / BLOCK_N≥32 这些「shape 约束」，根子其实在这块 TMEM + scale 布局上，而不在数学 shape 上。

④ 后面三步优化，各砍掉这条路的哪一跳

simple kernel 之所以「abysmal」(33–70 TFLOPS)，就是 scale 那条四跳的路太贵。三步优化逐段攻击它：
优化攻击的跳做法效果
优化 1 contiguoushop1改 GMEM 布局，让 [BLOCK_M, BLOCK_K//VEC] 块连续 → load 合并、不抖 L2~20×
优化 2 packed-block TMAhop1 + 部分 hop25D packed-block 布局 + 用 TMA 把 scale 直接搬进 SMEM，省掉大部分 convert_layout+~35%
优化 3 tcgen05_copyhop2 + hop3在 SMEM 里反 swizzle 后，用 tcgen05_copy 异步 SMEM → TMEM，完全绕开寄存器小幅
优化 4 pipeline / WS整条路并行化把「copy + mma」抽象成一条 async MMA，做软件流水线 / warp 专门化~2–2.4× 再翻倍
一句话串起来：operand 的搬运一开始就接近最优（TMA 直达 SMEM），全部 novelty 都在 scale 这条又长又绕的路上。
