PlanCTA 可视化：两个 CTA 如何分摊一个 matmul tile

这个页面只解释一个例子：matmul/sm90_num_ctas2 里的 TritonGPUPlanCTAPass。重点不是所有源码细节，而是看懂 before IR 到 after IR 之间的 compiler decision。

Kernelmatmul_kernel
Archcuda:90
num_ctas2

PassPlanCTA

1. 编译器在这里要回答的问题

输入事实

IR 已经有 tt.dot，并且 module 上写着 ttg.num-ctas = 2。

需要做决定

两个 CTA 共同处理一个 64x64 output tile 时，到底沿 M 维切，还是沿 N 维切？

输出 contract

把这个切分写进 CGALayout，让后续 layout cleanup、WGMMA lowering、store lowering 都能依赖。

2. 两种可能的 CTA 切法

原始 output tile C 的逻辑 shape 是 M x N = 64 x 64。
范围写法 [0,32) 表示 tile 内元素下标 0 到 31；右边界 32 不包含。
CTA 数量 总共有 2 个 CTA。某个维度没有切分时，两个 CTA 都覆盖这个维度的完整范围。
Before 中 dot parent 偏向的形态 CGALayout = [[1, 0]]
N dimension, size 64
N: original size 64, CTAs along N = 1, each CTA covers [0,64)
M dimension, size 64
M: CTA 0 covers [0,32), size 32M: CTA 1 covers [32,64), size 32
如果沿 M 维切：splitM=2, splitN=1 M 原始 size = 64，用 2 个 CTA，所以每个 CTA 负责 M size 32。 N 原始 size = 64，用 1 个 CTA，所以每个 CTA 都覆盖完整 N size 64。 CTA 0: C[M=[0,32), N=[0,64)]，shape = 32 x 64 CTA 1: C[M=[32,64), N=[0,64)]，shape = 32 x 64
→
PlanCTA 选出来的 after contract CGALayout = [[0, 1]]
N dimension, size 64
N: CTA 0 covers [0,32), size 32N: CTA 1 covers [32,64), size 32
M dimension, size 64
M: original size 64, CTAs along M = 1, each CTA covers [0,64)
PlanCTA 选择沿 N 维切：splitM=1, splitN=2 M 原始 size = 64，用 1 个 CTA，所以每个 CTA 都覆盖完整 M size 64。 N 原始 size = 64，用 2 个 CTA，所以每个 CTA 负责 N size 32。 CTA 0: C[M=[0,64), N=[0,32)]，shape = 64 x 32 CTA 1: C[M=[0,64), N=[32,64)]，shape = 64 x 32
CTA 0 负责的 output 子 tile CTA 1 负责的 output 子 tile

3. 为什么选 splitM=1, splitN=2

当前 dot shape

A: tensor<64x32xf16> B: tensor<32x64xf16> C: tensor<64x64xf32> numCTAs = 2

源码里的启发式

try chunk_m = 128, then 64 splitM = clamp(M / chunk_m, 1, numCTAs) splitN = numCTAs / splitM result: splitM = 1, splitN = 2

Value shape sizePerThread=[4,4] 表示一个 thread 持有 4x4 个 value；panel 里的每个 value cell 是其中一个逻辑元素，所以是 1x1。

5. IR 里最重要的 before / after 变化

Dot parent
Before: #blocked8 CGALayout = [[1, 0]]
After: #blocked CGALayout = [[0, 1]]
A/B operands
A/B convert to dot_op(parent=#blocked8) dot operand layout 跟旧 parent 绑定
A/B convert to dot_op(parent=#blocked) operand 跟新的 CTA split contract 绑定
Accumulator
#blocked → #blocked8 → #blocked dot 前后有 layout mismatch
#blocked → dot → #blocked loop-carried accumulator 和 dot result 对齐

6. 对照真实 IR

Before

#blocked8 = #ttg.blocked<{..., CGALayout = [[1, 0]]}>

%a_52 = ttg.convert_layout %a_50
  -> tensor<64x32xf16, #ttg.dot_op<{opIdx = 0, parent = #blocked8}>>

%b_53 = ttg.convert_layout %b_51
  -> tensor<32x64xf16, #ttg.dot_op<{opIdx = 1, parent = #blocked8}>>

%acc_54 = ttg.convert_layout %acc_49
  : tensor<64x64xf32, #blocked> -> tensor<64x64xf32, #blocked8>

%acc_55 = tt.dot ... -> tensor<64x64xf32, #blocked8>

After

#blocked = #ttg.blocked<{..., CGALayout = [[0, 1]]}>

%acc_42 = ttg.convert_layout %a_41
  -> tensor<64x32xf16, #ttg.dot_op<{opIdx = 0, parent = #blocked}>>

%acc_43 = ttg.convert_layout %b
  -> tensor<32x64xf16, #ttg.dot_op<{opIdx = 1, parent = #blocked}>>

%acc_44 = tt.dot %acc_42, %acc_43, %acc_40
  -> tensor<64x64xf32, #blocked>

7. 这个 pass 不是只改一行

第一步：在 dot 周围插入新的 layout contract

processDot 读取 dot 的 M/N/K 和 numCTAs， 构造新的 CGALayout、新的 accumulator layout，以及新的 A/B dot operand layout。

第二步：沿 producer / consumer 传播

insertCasts 后，PlanCTA 用队列把 casts 向前后传播到 loop、load/store、 broadcast、elementwise 等位置。这样 after IR 才会整体变得更一致。

8. 最短记忆版

PlanCTA 的问题： 多个 CTA 一起算一个 tensor tile 时，每个 CTA 到底负责哪一块？

这个例子的答案： 两个 CTA 沿 output 的 N 维切分，写成 CGALayout = [[0, 1]]。

为什么重要： 后续 pass 不再猜测 CTA ownership；它们可以直接依赖 dot operand/result 上已经固定的 layout contract。
