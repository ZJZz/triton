AssignLatencies 可视化

tt.latency 和 tt.self_latency 描述的是 stage distance

这个页面不讲硬件 cycle，也不讲精确性能模型。它只把一个更关键的概念画出来： numStages 是 software pipeline 的总 stage 预算， latency / self_latency 是在这个预算里给调度器的相对距离提示。

一句话 latency 不是周期数，而是 “producer 和 consumer 应该隔几拍 stage”。
最容易误解 tt.latency = 2 不是 load 需要 2 cycles，而是调度器尽量把 use chain 拉开 2 个 stage。
当前 pass 的角色 AssignLatencies 只写 hint；真正排到哪个 loop.stage 由 ScheduleLoops 决定。

先把三个量拆开

1. numStages

总预算

表示这个 loop 的 software pipeline 最多可以用多少个 stage 槽。 可以粗略理解成 “总逻辑节拍数”，但不是 GPU 硬件时钟。

例子：numStages = 3 时，调度器可用的槽通常就是 stage 0 / 1 / 2。

2. tt.latency = N

producer 到 users 的距离

表示某个 op 是 latency anchor，它到 downstream users 的 use chain 最好和它相隔 N 个 pipeline stage。

对 matmul load 来说，常见含义就是： “把 load 放早一点，把后面的 dot / MMA 放晚一点。”

3. tt.self_latency = N

op 自身相关 wait 的距离

主要用于 Blackwell MMAv5 一类 async MMA。它描述的不是 “这个 MMA 执行多久”，而是 “MMA 和它自己的 token / wait 关系可以推迟多少个 stage”。

这就是为什么它更像 “self wait distance”，而不是普通 producer-consumer latency。

token 和 async wait 是什么

token = 依赖凭证

在 Blackwell MMAv5 路径里，!ttg.async.token 不是数据值， 而是 “某个异步动作已经完成到什么程度” 的 SSA 凭证。

acc_tok → mma_tok → load_tok

例如 tmem_alloc 产出 acc_tok， tc_gen5_mma 消费它并产出 mma_tok， 后续 tmem_load 再消费 mma_tok。

async wait = 把依赖兑现成等待

token 只是高层 IR 的依赖表达。后续 lowering 会把这种依赖变成更具体的 wait / barrier / token-consume 位置，让结果在真正被读取前完成同步。

launch MMA → do other stage work → wait / tmem_load

所以 tt.self_latency 说的不是 “MMA 执行多久”，而是 “这条 MMA 自身相关的 token / wait 链能往后推几拍 stage”。

一句话：token 是依赖边，wait 是后续 lowering 用来兑现这条依赖边的同步动作。

两个固定例子

场景 典型值 它在描述什么 不该怎么理解
Ampere / Hopper 的 direct load path numStages = 3
tt.latency = 2		 load 是 anchor。调度器会尽量把 load 放到更早 stage，把 convert_layout / local_alloc / dot 放到更晚 stage。 不是 “global load 物理上只需要 2 cycles”。
Blackwell MMAv5 path tt.latency = 1
tt.self_latency = 1		 MMA 的 users 可以被推迟 1 个 stage；同时 MMA 自己的 async token / wait 关系 也可以推迟 1 个 stage。 不是 “MMA 指令本身的真实执行延迟是 1”。
这也是为什么 note 里说它更像 “stage dependency distance hint”。 live range 会受它影响，但 live range 是调度结果，不是这个 attr 自身的定义。

最后的边界

不是 cycle 模型

`AssignLatencies` 没有在这里精确建模 global memory latency、 issue rate、occupancy、register pressure 或 bank conflict。

不是最优性证明

这个数值不能保证 pipeline 全局最优。它只是在现有 numStages 预算下，给后续调度器一个保守但可执行的距离约束。

它真正的价值

没有这个数值，ScheduleLoops 只能看到普通 SSA def-use， 却不知道哪条链应该跨迭代提前。这个 attr 把 “值得提前的 op” 显式标出来了。
