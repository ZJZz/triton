TMEM 这条线要拆成四个问题：能不能进、进去后怎么排、物理放哪儿、什么时候必须停一下

这里最容易乱的地方，是把 placement、layout、allocation、synchronization 全部混成“TMEM 优化”。 这个页面按 contract 拆开：先决定值能不能从 shared path 切到 TMEM，再看局部 layout rewrite， 最后才是 row/col 分配和 barrier 插入。

placement

PromoteLHSToTMem 决定某些 TCGen5 MMA 的 A operand 是否从 shared 改为 TMEM。

allocation

TensorMemoryAllocation 给抽象 ttng.tmem_alloc 写入 tensor_memory_col_offset 和 row_offset。

barrier

TMemBarrierInsertion 不为“所有 TMEM 访问”加 barrier，而是看 WAR / RAW / WAW 和 load/store -> mma 依赖。

主视图

切换一个问题一个问题地看，不要试图一次把 TMEM 全部细节塞进脑子里。

Placement：什么样的 LHS 才允许切到 TMEM

这里的目标不是“尽量多用 TMEM”，而是只在 contract 明确、layout 可落地时才把 A operand 改成 TMEM memdesc。

constraint 1 源必须来自 ttg.local_alloc

不是所有 SSA value 都能直接变成 TMEM；pass 先要求它已经是 local/shared materialization 过的对象。

constraint 2 local_alloc 和 tc_gen5_mma 必须在同一 region

源码注释写得很直白：要把 TMEM allocation 的生命期限制在单个 block 内。

constraint 3 element bit width 只能是 8 / 16 / 32，且不能是 fp4 padded

超出这个集合就直接失败，不是“先放进来再想办法修”。

decision layout 不兼容时，只有部分来源才允许插 convert_layout

对来自 load / block-arg 的来源更保守；除非环境变量放开，否则宁可不 promote。

Barrier：不是“所有 TMEM op 都停一下”，而是看真实依赖

用户已经对源码核对过：WAR / RAW / WAW 都算，load -> mma 和 store -> mma 也要 barrier；mma -> load/store 不需要，因为后续有 mbarrier wait。

must barrier WAR / RAW / WAW on TMEM object

同一块 TMEM 物理对象只要存在真实别名依赖，就必须把顺序固定住。

must barrier tmem_load -> tc_gen5_mma / tmem_store -> tc_gen5_mma

这是“先访问 TMEM，再让 MMA 以 async 方式继续使用它”的典型边界。

no extra barrier tc_gen5_mma -> tmem_load/store

这类顺序由后续的 mbarrier wait 覆盖，不需要再额外补一条 CTA-level barrier。

observed in dump 当前 matmul dump 在首个 MMA 前插入了 ttg.barrier local

你可以把它理解成“开始把 shared/TMEM 状态交给第一个 async MMA 之前，先把可见性和顺序收紧”。

阅读提示

这条路径的难点不在“记住 pass 名字”，而在于知道每个 pass 正在建立哪一种 contract。

推荐心智顺序
  • 先问这个 value 能不能进 TMEM，而不是先问物理地址。
  • 再问 TMEM 边界是否需要改 layout，避免在错误层面追踪 reshape / join。
  • 最后才看 allocator 和 barrier，因为这两者依赖前面的 placement 与 use pattern。
和 TMA 的边界

TMA 的主问题是 global <-> shared transport。TMEM 的主问题是 Blackwell tensor-core state 该不该驻留在 TMEM，以及之后怎样同步和分配。
