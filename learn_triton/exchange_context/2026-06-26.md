## 已确认的重要技术结论

### 1. `ConvertTritonToTritonGPU`

不是单纯“给 tensor 加 layout”。

更准确地说：

- 它把抽象的 `Triton IR` 转成带 GPU 执行语义的 `TTGIR`
- 让 IR 开始知道 tensor 将如何映射到 thread / warp / CTA

关键参考：

- [019_After_ConvertTritonToTritonGPU.mlir:2](/LocalRun/jiangzhe.zhao/my_repo/triton/learn_triton/dump/vecadd/mlir-pass-dump.split/019_After_ConvertTritonToTritonGPU.mlir:2)
- [019_After_ConvertTritonToTritonGPU.mlir:8](/LocalRun/jiangzhe.zhao/my_repo/triton/learn_triton/dump/vecadd/mlir-pass-dump.split/019_After_ConvertTritonToTritonGPU.mlir:8)

### 2. `TritonGPUCoalesce`

主要针对 `global memory` 访问模式。

更准确地说：

- 在 `TTGIR` 里改写 layout / encoding
- 常常会先引入一些 `ttg.convert_layout`
- 后续再由 `RemoveLayoutConversions` 从更全局的角度决定哪些能消掉

### 3. `RemoveLayoutConversions`

不是单纯 cleanup。

更准确地说：

- 它是 layout unification / rematerialization pass
- 在 `vecadd` 里，最后把链条统一到更适合 coalesced access 的 layout，而不是机械回退到原始 layout

关键参考：

- Before: [026_Before_TritonGPURemoveLayoutConversions.mlir](/LocalRun/jiangzhe.zhao/my_repo/triton/learn_triton/dump/vecadd/mlir-pass-dump.split/026_Before_TritonGPURemoveLayoutConversions.mlir)
- After: [027_After_TritonGPURemoveLayoutConversions.mlir](/LocalRun/jiangzhe.zhao/my_repo/triton/learn_triton/dump/vecadd/mlir-pass-dump.split/027_After_TritonGPURemoveLayoutConversions.mlir)

## 环境和脚本上的重要上下文

有两个环境相关问题之前已经处理过，换机器继续时要注意：

1. `learn_triton/compile_driver.py` 之前有一个实际问题：

- 它原来调用了 `kernel.create_binder()`
- 这会触发 Triton runtime driver detection
- 当前环境下会报 `0 active drivers`

后来已经改成直接使用 `ASTSource` from `triton.compiler`，避免依赖 runtime driver。

2. 重新生成 dump 时，需要设置：

```bash
TRITON_CACHE_DIR=/tmp/triton-cache
```

否则 Triton 可能尝试写：

```bash
/home/jiangzhe.zhao/.triton/cache
```

在当前环境里那是只读路径，会失败。
