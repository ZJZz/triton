# 2026-06-26 学习笔记

## 今天的目标

今天主要澄清了一件很重要的事：

- 我们看的 pass，不应该只问“它对应哪一种 CUDA optimization”
- 还要问“这是 Triton 自己在做什么”

也就是说，分析一个 pass 时，最好同时带着两个问题：

1. `Triton mechanism`：这个 pass 在 Triton 编译器内部，具体承担什么职责？
2. `optimization goal`：这个 pass 服务的性能目标是什么？它是不是在落实某种 GPU / CUDA 风格的 optimization？

这两个维度分开后，整个 backend pipeline 会清楚很多。

## 今天形成的核心理解

### 1. 不要把所有 pass 都当成 “CUDA optimization pass”

很多 pass 的本质并不是直接做 CUDA optimization，而是：

- 把高层 Triton IR 变成更接近 GPU 执行模型的 IR
- 给 tensor 加 layout
- 引入/消除 Triton 特有的中间表示
- 为后续 lowering 和优化创造条件

所以很多 pass 更准确地说是：

- `Triton 提供 mechanism`
- `optimization 提供 reason`

例如：

- `TritonGPUCoalesce`
  - Triton 自己在做的事：重写 TTGIR 中的 layout / encoding
  - 它服务的目标：coalesced memory access

这类 pass 不能简单归类成“纯 CUDA”或者“纯 Triton”，而是要拆成这两个层次看。

### 2. TTGIR 是最值得重点学习的阶段

目前对 Triton backend 的理解可以先抓这条主线：

- `TTIR`：表达“要做什么 tensor 计算”
- `TTGIR`：表达“这些 tensor 计算如何映射到 GPU thread / warp / CTA / memory layout 上”
- `LLVM / PTX / cubin`：把前面已经决定好的执行策略，继续 lower 成更底层表示

所以如果团队重点关注：

- `TTGIR`
- `Warp Specialization`
- `tensor memory`

那其实是在关注 Triton 最核心、最“有编译器味道”的部分。

### 3. 学 pass 时，最好把 Triton 自身机制和硬件优化目标分开记

后面看 dump 时，建议每个 pass 都记两列：

- 这一列写：它在 Triton IR 层面改了什么
- 另一列写：它想达成什么性能收益

这样就不会把“IR 结构变化”和“性能意图”混在一起。

## Study Table

下面这张表是今天最值得保留的结论。

### TTGIR Core Passes

| Pass | Triton mechanism | optimization goal |
|---|---|---|
| `ConvertTritonToTritonGPU` | 把抽象的 Triton tensor IR 转成带 GPU encoding 的 TTGIR，比如 `#blocked`、warp/CTA 结构、GPU-aware tensor type | 建立 data-to-thread mapping，为后续所有 GPU optimization 打基础 |
| `TritonGPUCoalesce` | 调整 TTGIR 中的 layout / encoding，让访问模式更适合 GPU memory access | 实现 coalesced global memory access |
| `PlanCTA` | 在 Triton IR 里决定 CTA 级别的数据/工作分配方式 | 提升 block-level locality 和执行组织效果 |
| `RemoveLayoutConversions` | 消除 Triton 自己引入但后来变得多余的 `convert_layout` | 减少额外数据搬运、register pressure 和 shuffle 开销 |
| `OptimizeThreadLocality` | 调整 Triton 中每个 thread 持有/处理的数据形态 | 提升 thread-local locality，降低每线程低效访问 |
| `OptimizeDescriptorEncoding` | 调整 tensor descriptor 相关 encoding | 提升 layout 兼容性，减少 descriptor 使用成本 |

### Matmul / Tensor Core Passes

| Pass | Triton mechanism | optimization goal |
|---|---|---|
| `AccelerateMatmul` | 把 Triton 的 `tt.dot` / matmul 形式改写成更适合 MMA 的 TTGIR 形式 | 更高效使用 tensor cores |
| `OptimizeDotOperands` | 调整 dot operand 的 layout / shape / encoding | 更高效喂给 tensor cores，减少 operand conversion 成本 |
| `LowerMMA` | 把 Triton / NVIDIA MMA 抽象继续 lower 到更具体 backend op | 把高层 matmul 映射到 `mma.sync` / `wgmma` 等硬件执行路径 |

### Pipeline / Latency-Hiding Passes

| Pass | Triton mechanism | optimization goal |
|---|---|---|
| `AssignLatencies` | 在 Triton loop pipeline 模型中附加 latency 信息 | 帮助编译器理解哪些 op 应该 overlap |
| `ScheduleLoops` | 在 TTGIR 中重排 / 分阶段组织 loop 内操作 | 隐藏 latency，形成 producer/consumer 风格结构 |
| `Pipeline` | 构建 Triton 自己的软件 pipeline loop 结构 | overlap memory and compute |
| `Prefetch` | 在 Triton IR 中提前安排数据移动 | 隐藏 memory latency |

### Warp Specialization Passes

| Pass | Triton mechanism | optimization goal |
|---|---|---|
| `WarpSpecialize` / `NVGPUWarpSpecialization` | 在 Triton IR 中显式引入 specialized warp role，而不是所有 warp 做对称工作 | producer/consumer warp specialization，提高 overlap 和吞吐 |
| `OptimizePartitionWarps` | 重新分配 specialized partition 使用多少 warps | 更好平衡 register 使用和 partition 效率 |
| `ConvertWarpSpecializeToLLVM` | 把 Triton 中的 warp specialization 构造 lower 掉 | 在更低层代码中保留 specialization 的收益 |

### Tensor Memory Passes

| Pass | Triton mechanism | optimization goal |
|---|---|---|
| `PromoteLhsToTMEM` | 把某些 Triton operand / accumulator 转成更偏 TMEM 的表示 | 利用 hardware tensor memory 路径 |
| `HoistTMEMAlloc` | 调整 TMEM alloc 在 Triton IR 中的生命周期和位置 | 降低 lifetime 成本，支持 pipelining，避免不良放置 |
| `OptimizeTMemLayouts` | 优化 TMEM 的 layout / encoding | 提升 TMEM access 效率 |
| `InterleaveTMEM` | 重排 TMEM 使用或布局方式 | 匹配硬件访问模式，提升吞吐 |
| `AllocateTensorMemory` | 在 lowering 过程中把 TMEM 的使用具体化成 allocation | 让 tensor memory 变成真实、可落地的资源分配 |
| `TMEMBarrierInsertion` | 给 Triton 的 TMEM 相关操作插入同步 | 保证 async / partitioned TMEM usage 的正确性 |
| `RemoveTMEMTokens` | 清理 Triton 内部为了表达 TMEM 依赖而引入的 token 机制 | 在依赖关系稳定后简化 IR |

### Boundary / Lowering Passes

| Pass | Triton mechanism | optimization goal |
|---|---|---|
| `AllocateSharedMemoryNv` | 把 Triton 中对 shared memory 的需求变成具体分配和布局 | 更高效使用 shared memory |
| `ConvertTritonGPUToLLVM` | 把 TTGIR lower 到 LLVM dialect，去掉 Triton/TritonGPU op | 把上游的 layout / scheduling / memory 决策真正落地 |
| `ConvertNVGPUToLLVM` | 继续 lower NVIDIA-specific IR | 为 backend codegen 做准备 |
| `ConvertNVVMToLLVM` | 最后的 dialect conversion | 完成 codegen 入口准备 |

## 今天对 “Triton 自己在做什么” 的理解

今天比较明确的一点是：Triton 不只是“把 Python kernel 翻译成 CUDA”。

它在 backend 里自己做了很多事情：

- 定义自己的 IR 层次：`TTIR`、`TTGIR`
- 定义自己的 layout system：`#blocked`、`#mma`、shared/TMEM encodings
- 自动决定 tensor 到 thread/warp/CTA 的映射
- 自动插入和消除 layout conversion
- 自动构建 pipeline / scheduling 结构
- 在新架构上引入 warp specialization 相关中间表示
- 为 tensor memory 设计专门的 IR、allocation、layout、barrier 处理

也就是说，很多“optimization”其实是通过 Triton 自己的抽象系统表达出来的。

## 今天对 “CUDA optimization” 的理解

CUDA / GPU optimization 更像是目标，而不是唯一的实现方式。

这些目标包括：

- coalesced global memory access
- shared memory reuse
- tensor core usage
- latency hiding
- overlap memory and compute
- warp role specialization
- synchronization correctness
- register / shared memory / occupancy tradeoff

Triton 的 pass 往往是在用自己的 IR 机制去实现这些目标。

## 一个很有用的判断方法

以后看一个 pass，可以先判断它更偏哪一类：

### 1. Triton semantic / lowering pass

重点看：

- 是否在引入或消除 Triton 特有 IR
- 是否在推进 IR 阶段切换

典型例子：

- `ConvertTritonToTritonGPU`
- `ConvertTritonGPUToLLVM`

### 2. Triton layout-management pass

重点看：

- 是否在调整 encoding、layout、descriptor、TMEM layout

典型例子：

- `TritonGPUCoalesce`
- `RemoveLayoutConversions`
- `OptimizeDotOperands`
- `OptimizeTMemLayouts`

### 3. Hardware optimization pass

重点看：

- 是否在做 latency hiding、tensor core acceleration、warp role partition、pipeline

典型例子：

- `AccelerateMatmul`
- `AssignLatencies`
- `ScheduleLoops`
- `Pipeline`
- `WarpSpecialize`

### 4. Cleanup / support pass

重点看：

- 是否只是清理、合并、删死代码、简化表达

典型例子：

- `Canonicalizer`
- `CSE`
- `SymbolDCE`

### 2. After：新增 `#blocked`，tensor 开始带 GPU layout

最关键的变化先看这一行：

- [019_After_ConvertTritonToTritonGPU.mlir:2](/LocalRun/jiangzhe.zhao/my_repo/triton/learn_triton/dump/vecadd/mlir-pass-dump.split/019_After_ConvertTritonToTritonGPU.mlir:2)
  定义了 `#blocked = #ttg.blocked<{sizePerThread = [1], threadsPerWarp = [32], warpsPerCTA = [4], order = [0]}>`

这行非常关键，因为它第一次把 GPU layout 明确写进 IR。

可以先这样理解：

- `sizePerThread = [1]`：每个 thread 持有多少元素
- `threadsPerWarp = [32]`：一个 warp 用 32 个线程
- `warpsPerCTA = [4]`：一个 CTA 里有 4 个 warps
- `order = [0]`：这一维的线性顺序

这个理解先够用，后面学 `Coalesce` 和更复杂 layout 时再细化。

### 3. After：module 级别也开始带 GPU 执行属性

除了 tensor layout，module 自己也变了：

这里新增了：

- `ttg.num-ctas = 1`
- `ttg.num-warps = 4`
- `ttg.target = "cuda:86"`

这说明这个 pass 不只是“给 tensor 加 layout”，它还让整个 IR 进入了 GPU-aware 的执行上下文。

所以更准确的说法是：

- Before：IR 知道“要算什么”
- After：IR 开始知道“要按什么 GPU 执行模型去算”

### 5. 这一节最值得记住的结论

如果以后再看到 `ConvertTritonToTritonGPU`，可以直接这样记：

- 它当然会“给 tensor 加 layout”
- 但本质上不只是加 layout
- 它是在把 `Triton IR` 转成 `TTGIR`
- 让 IR 知道目标 GPU、warp/CTA 结构、以及 tensor 到执行层次的映射方式

也可以再压缩成一句：

- `ConvertTritonToTritonGPU` 的核心作用，是让 IR 从 “what to compute” 进入 “how GPU will organize and execute it”

### 6. 它属于哪一类 pass

按我们今天的分类，它更偏：

- `Triton semantic / lowering pass`

而不是一开始就直接做性能优化的 pass。

它的主要作用是：

- 建立 TTGIR 的 GPU 语义基础
- 为后续 `TritonGPUCoalesce`、`RemoveLayoutConversions` 等 pass 提供 layout 基础

所以它更像：

- `Triton mechanism` 很强
- `optimization` 还处在“铺路阶段”

### 4. 这一节最值得记住的结论

可以把 `Coalesce` 先稳定记成下面三句：

- `TritonGPUCoalesce` 主要是在 TTGIR 里优化 global memory 访存布局
- 它的优化目标和 GPU 硬件有关，但实现并不是简单按具体 arch 硬编码分叉
- 它常常会先引入一些 `ttg.convert_layout`，后面再交给 `RemoveLayoutConversions` 去消掉不必要的 conversion

如果和前一个 pass 连起来看：

1. `ConvertTritonToTritonGPU`
   先让 IR 带上 GPU 执行语义和基础 layout
2. `TritonGPUCoalesce`
   再针对 global memory access 调整更适合访存的 layout
3. `RemoveLayoutConversions`
   最后尝试把前面引入但已经不必要的 `convert_layout` 清掉

这三个 pass 连起来，就是我们读 `vecadd` 时最清楚的一条主线。

### 3. 为什么这不是“前一个 pass 白做了”

这里最容易产生的误解是：

- 上一个 pass 明明加了很多 `convert_layout`
- 这个 pass 又把它们都消掉了
- 那是不是前面的事都白做了

目前基于 `vecadd` 的理解，答案是否定的。

更准确的理解是：

- `TritonGPUCoalesce` 的职责是先把 memory op 附近改成更适合 coalesced global memory access 的 layout
- 这一步是局部、保守、明确的，所以它会先显式插入 `ttg.convert_layout`
- `RemoveLayoutConversions` 再站在更全局的视角问：既然访存已经偏向这个 layout，周围那段计算链能不能一起改成同一个 layout

所以：

- `Coalesce` 负责“把更好的访存 layout 提出来”
- `RemoveLayoutConversions` 负责“尽量把整段链条并到这个 layout 上”

两者不是重复，而是前后衔接。

### 4. 为什么最后统一到 `sizePerThread = [4]`，而不是退回 `[1]`

对这个例子，当前最实用的理解是：

- `Coalesce` 已经证明对这个 global memory access 来说，`sizePerThread = [4]` 更适合作为访存 layout
- `RemoveLayoutConversions` 再看整条链时，发现中间只有很便宜的 elementwise 计算，也就是一个 `arith.addf`
- 这时最划算的做法，不是把 load/store 再拉回旧 layout，而是把这条很短的算术链也一并改写到 `[4]`

所以最终结果是：

- 访存收益保住了
- conversion 消失了
- 中间计算也没有变复杂

这正是一个“把局部 layout 优化扩展成整条链统一 layout”的例子。
