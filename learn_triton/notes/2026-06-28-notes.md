`PlanCTA` 回答的问题是：

```text
多个 CTA 一起处理一个 logical tensor tile 时，每个 CTA 到底负责哪一块？
```

当前例子里：

```text
C tile = 64 x 64
num_ctas = 2
```

所以 compiler 必须决定两个 CTA 是沿 `M` 切，还是沿 `N` 切。这个决定最后会写进
`CGALayout`，成为后续 layout cleanup、MMA/WGMMA lowering、store lowering 能依赖的
contract。

## 2. 核心结论

一句话：

```text
PlanCTA 在这个例子里以 dot result / accumulator tile 为锚点，
选择 splitM=1, splitN=2，也就是两个 CTA 沿 output N 维切 C[64,64]。
```

Before IR 里 dot parent 是：

```text
#blocked8
CGALayout = [[1, 0]]
```

这对应一种沿 `M` 维切的形态：

```text
CTA 0: C[M [0,32),  N [0,64)]  shape = 32 x 64
CTA 1: C[M [32,64), N [0,64)]  shape = 32 x 64
```

After IR 里 PlanCTA 选择的新 dot parent 是：

```text
#blocked
CGALayout = [[0, 1]]
```

也就是沿 `N` 维切：

```text
CTA 0: C[M [0,64), N [0,32)]   shape = 64 x 32
CTA 1: C[M [0,64), N [32,64)]  shape = 64 x 32
```

需要特别注意：`PlanCTA` 处理 `tt.dot` 时，核心 tiling decision 是从 dot 的结果 /
accumulator tile 出发的，不是先从 A/B operand 出发。

源码里的数据来源是：

```text
dTy = dot.getD().getType()
M = dTy.getShape()[0]
N = dTy.getShape()[1]
K = aTy.getShape()[1]
splitM, splitN = getCTATiling(M, N, K, getNumCTAs(dLayout))
```

也就是说：

- `M/N` 来自 dot 的 `D` operand，也就是 accumulator / result tile：`C[M,N]`
- `K` 来自 A operand 的第 1 维：`A[M,K]`
- CTA split 只生成 `{splitM, splitN}`，也就是对 output tile 的 M/N 维做切分

然后源码先构造新的 result parent layout：

```text
newDLayout = BlockedEncodingAttr::get(..., newCGALayout)
```

再从这个 result parent layout 派生 A/B operand layout：

```text
newALayout = DotOperandEncodingAttr::get(opIdx = 0, parent = newDLayout)
newBLayout = DotOperandEncodingAttr::get(opIdx = 1, parent = newDLayout)
```

所以顺序可以理解成：

```text
先决定 C / accumulator / dot result 的 CTA ownership
  -> 再从 result parent layout 派生 A/B dot operand layout
  -> 最后通过 cast propagation 传播到 A/B load 和 pointer prologue
```

## 3. 为什么选 splitM=1, splitN=2

当前 dot shape 是：

```text
A: tensor<64x32xf16>
B: tensor<32x64xf16>
C: tensor<64x64xf32>
numCTAs = 2
```

代入 `processDot` / `getCTATiling`：

```text
M = 64
N = 64
K = 32
numCTAs = 2
```

当前源码的思路可以简化成：

```text
for chunk_m = 128, 64 while chunk_m >= 64:
  splitM = clamp(M / chunk_m, 1, numCTAs)
  splitN = numCTAs / splitM
  if N / splitN >= 64:
    break
```

对这个输入：

```text
chunk_m = 128:
  splitM = clamp(64 / 128, 1, 2) = 1
  splitN = 2 / 1 = 2
  N / splitN = 64 / 2 = 32 < 64, no break

chunk_m = 64:
  splitM = clamp(64 / 64, 1, 2) = 1
  splitN = 2 / 1 = 2
  N / splitN = 64 / 2 = 32 < 64, no break

next chunk_m = 32 is illegal, loop exits
```

所以最后保留上一轮结果：

```text
splitM = 1
splitN = 2
```

`K` 会从 dot type 中读出来，并作为参数传进 `getCTATiling`，但当前
`getCTATiling` 的函数体没有使用 `K`。也就是说，在当前实现里这个 tiling 选择只由
`M/N/numCTAs` 决定。

## 4. After #blocked layout 的分层

After IR 的主 layout 是：

```text
#blocked = #ttg.blocked<{
  sizePerThread = [4, 4],
  threadsPerWarp = [4, 8],
  warpsPerCTA = [4, 1],
  order = [1, 0],
  CGALayout = [[0, 1]]
}>
```

可以按四层理解：

```text
C[64,64]
  -> CTA 层：CGALayout = [[0,1]]
  -> Warp 层：warpsPerCTA = [4,1]
  -> Thread 层：threadsPerWarp = [4,8]
  -> Value 层：sizePerThread = [4,4]
```

`threadsPerWarp = [4,8]` 不是手工指定的新常量，而是 `processDot` 里构造
`newDLayout` 时由 `BlockedEncodingAttr::get` 重新推导出来的。它保留旧 dot parent
的 `sizePerThread = [4,4]` 和 `warpsPerCTA = [4,1]`，但 CTA tile 从 M-split 的
`32x64` 变成 N-split 的 `64x32`，因此 thread 分配从 `[2,16]` 变成 `[4,8]`。

CTA 层：

```text
CGALayout = [[0,1]]
M size = 64
N size = 64 / 2 = 32
CTA shape = 64 x 32

CTA 0: C[M [0,64), N [0,32)]
CTA 1: C[M [0,64), N [32,64)]
```

Warp 层：

```text
warpsPerCTA = [4,1]
M size = 64 / 4 = 16
N size = 32 / 1 = 32
warp shape = 16 x 32

CTA 0 / warp 0: C[M [0,16),  N [0,32)]
CTA 0 / warp 1: C[M [16,32), N [0,32)]
CTA 0 / warp 2: C[M [32,48), N [0,32)]
CTA 0 / warp 3: C[M [48,64), N [0,32)]
```

Thread 层：

```text
threadsPerWarp = [4,8]
M size = 16 / 4 = 4
N size = 32 / 8 = 4
thread shape = 4 x 4
```

对 `CTA 0 / warp 0`：

```text
thread id = t
thread_m_group = floor(t / 8)
thread_n_group = t % 8

M start = warpMStart + thread_m_group * 4
N start = ctaNStart + thread_n_group * 4

T0: C[M [0,4),  N [0,4)]
T1: C[M [0,4),  N [4,8)]
T8: C[M [4,8),  N [0,4)]
```

Value 层：

```text
sizePerThread = [4,4]
```

这表示一个 thread 持有一个 `4 x 4` 的 logical element block。可视化面板里的每个
`value cell` 是这个 block 里的一个 single logical element：

```text
value shape = 1 x 1

value id = v
value_m = threadMStart + floor(v / 4)
value_n = threadNStart + (v % 4)

v0:  C[M 0, N 0]
v1:  C[M 0, N 1]
v4:  C[M 1, N 0]
v15: C[M 3, N 3]
```

## 5. Pass 机制：diff 是怎么生成的

一句话总览：

```text
processDot 选新的 CTA 切分 splitM=1, splitN=2
  -> 在 tt.dot 周围插入 UnrealizedConversionCast
  -> backward cast 往 producer 方向传播
  -> forward cast 往 consumer 方向传播
  -> 传播过程中重写 op 的 tensor encoding
  -> 相邻 cast 被抵消，或落成真正需要的 ttg.convert_layout
```

本例的运行路径来自 `CTAPlanner::run`：

```text
processDot(funcOp)
processReduce(funcOp)
if (!tiled) processStoreLikeOps(funcOp)
while (!queue.empty())
  propagate(cast)
```

当前 kernel 有 `tt.dot`，没有 `tt.reduce`。`processDot` 找到 dot 后调用 `markTiled()`，
所以 `processStoreLikeOps` 不作为 tiling 锚点运行。store 后面仍然会被改，是因为 dot
周围插入的 cast 继续向 consumer 方向传播到了 trunc/store。

这些 diff 不是一组逐行替换规则，而是一个 layout decision 沿数据流传播后的结果。
每一处变化基本都属于两类之一：

- 新的 `CGALayout` / tensor encoding 本身
- cast propagation、cast 消除、producer clone 产生的副产物

`insertCasts` 给 dot 的 A/B/acc operands 插 backward cast，给 dot result 插 forward
cast。后续传播时：

- backward cast 往 producer 方向走，比如 load、addptr、broadcast、make_range、constant
- forward cast 往 consumer 方向走，比如 `scf.yield`、`scf.for` result、trunc、store
- 如果遇到已有的 `ttg.convert_layout`，可能被 backward/forward 传播吃掉
- 如果临时 `UnrealizedConversionCastOp` 相邻且类型相同，`eliminateAdjacentCasts`
  会 `replaceAllUsesWith`，两个 cast 直接消失
- 如果两端类型不同，最后才落成真正需要的 `ttg.convert_layout`

所以 prologue 里 `convert_layout` 大量减少、layout 定义从 9 个降到 5 个，是传播和
cast 抵消的副产物，不是这个 pass 单独追求的目标。

注意：`#loc` 编号、SSA 名字、`#blockedN` 别名编号的变化大多是 MLIR printer 重新编号，
不是 PlanCTA 的核心语义变化。源码里 pass 结束前还会 clone func 再 erase 旧 func，
这是为了让 dump 能显示 pass 后变化。

### 6.3 accumulator 常量和 loop iter_arg 跟着变

Before：


After：

这里容易被 `#blocked` 名字误导：Before 和 After 的 `#blocked` 不是同一个 layout。

它必须同时改四个位置：

### 6.5 prologue 指针计算里的 convert_layout 大量减少

Before 的 A/B pointer prologue 里有很多中间 layout：

```text
#blocked1
#blocked2
#blocked3
#blocked4
#blocked5
#blocked6
#blocked7
```

After 收敛成少数几个 layout：

```text
#blocked1  // N-split, store/B-friendly
#blocked2  // A/K-side layout
#blocked3  // N-side slice/broadcast path
#blocked4  // A load replicated layout
```

生成机制是 cast propagation：

```text
make_range       -> processMakeRange
splat            -> processSplat
expand_dims      -> processExpandDimsBackward
broadcast        -> processBroadcast
addptr / arith   -> processElementwise
load/store       -> processLoadStore
convert_layout   -> processConvertLayoutBackward/Forward
```

每个 producer 如果只有一个 use，pass 通常可以直接把它重写成目标 layout。
如果传播后产生相邻 cast，same type 会直接抵消，different type 才会生成一个真实的
`ttg.convert_layout`。

## 7. Compiler contract

Input contract：

- module 上已经有 `ttg.num-ctas = 2`
- IR 里已经有 `tt.dot`
- dot operand/result 已经有 TTGIR layout encoding

Output contract：

- dot parent layout 明确表达 CTA ownership
- A/B dot operand parent 和 accumulator/result 使用一致的 CTA split contract
- producer/consumer 链尽量传播同一个 CTA ownership model

Invariant：

- tensor logical shape 不变：A 仍是 `64x32`，B 仍是 `32x64`，C 仍是 `64x64`
- element type 不变：A/B 是 `f16`，accumulator 是 `f32`
- 数学语义不变：还是同一个 matmul tile
- 改变的是 layout / encoding / CTA ownership / layout conversions

如果没有 PlanCTA：

- `num_ctas=2` 不会被系统性落实到 dot 的 accumulator layout 上
- dot 可能继续使用 before IR 里旧的临时 parent layout
- dot 前后会保留更多 `ttg.convert_layout`
- 后续 RemoveLayoutConversions、MMA/WGMMA lowering、store lowering 不知道应该依赖哪一种 CTA ownership

最重要的是，IR 里会缺少一个清楚的答案：

```text
这两个 CTA 到底沿 M 还是沿 N 分摊输出 tile？
```

当前例子的答案就是：

```text
两个 CTA 沿 output N 维切分。
CGALayout = [[0, 1]]
CTA 0 shape = 64 x 32
CTA 1 shape = 64 x 32
```
