## 核心问题

`PlanCTA` 回答的问题是：

```text
多个 CTA 一起处理一个 logical tensor tile 时，每个 CTA 到底负责哪一块？
```

当前例子里：

```text
C tile = 64 x 64
num_ctas = 2
```

所以 compiler 必须决定两个 CTA 是沿 `M` 切，还是沿 `N` 切。这个决定最后会写进 `CGALayout`，成为后续 layout cleanup、MMA/WGMMA lowering、store lowering 能依赖的 contract。

## 当前例子的 PlanCTA 决策

Before IR 里 dot parent 是：

```text
#blocked8
CGALayout = [[1, 0]]
```

这对应一种沿 `M` 维切的形态：

```text
CTA 0: C[M [0,32),  N [0,64)]  shape = 32 x 64
CTA 1: C[M [32,64), N [0,64)]  shape = 32 x 64
```

After IR 里 PlanCTA 选择的新 dot parent 是：

```text
#blocked
CGALayout = [[0, 1]]
```

也就是沿 `N` 维切：

```text
CTA 0: C[M [0,64), N [0,32)]   shape = 64 x 32
CTA 1: C[M [0,64), N [32,64)]  shape = 64 x 32
```

一句话总结：

```text
PlanCTA 在这个例子里选择 splitM=1, splitN=2。
```

## 为什么选 splitM=1, splitN=2

当前 dot shape 是：

```text
A: tensor<64x32xf16>
B: tensor<32x64xf16>
C: tensor<64x64xf32>
numCTAs = 2
```

代入 `processDot` / `getCTATiling`：

```text
M = 64
N = 64
K = 32
numCTAs = 2
```

当前源码的思路可以简化成：

```text
try chunk_m = 128, then 64
splitM = clamp(M / chunk_m, 1, numCTAs)
splitN = numCTAs / splitM
```

对这个输入，最后得到：

```text
splitM = 1
splitN = 2
```

注意：`K` 会从 dot type 中读出来，但这个 tiling 选择主要由 `M/N/numCTAs` 决定。

## After #blocked layout 的分层

After IR 的主 layout 是：

```text
#blocked = #ttg.blocked<{
  sizePerThread = [4, 4],
  threadsPerWarp = [4, 8],
  warpsPerCTA = [4, 1],
  order = [1, 0],
  CGALayout = [[0, 1]]
}>
```

可以按四层理解：

```text
C[64,64]
  -> CTA 层：CGALayout = [[0,1]]
  -> Warp 层：warpsPerCTA = [4,1]
  -> Thread 层：threadsPerWarp = [4,8]
  -> Value 层：sizePerThread = [4,4]
```

### CTA shape

`CGALayout = [[0,1]]` 表示 `M` 不切，`N` 用 2 个 CTA 切：

```text
M size = 64
N size = 64 / 2 = 32

CTA shape = 64 x 32
```

范围：

```text
CTA 0: C[M [0,64), N [0,32)]
CTA 1: C[M [0,64), N [32,64)]
```

### Warp shape

`warpsPerCTA = [4,1]` 表示每个 CTA 内部沿 `M` 方向分成 4 条 warp band，`N` 方向不再切：

```text
M size = 64 / 4 = 16
N size = 32 / 1 = 32

warp shape = 16 x 32
```

范围例子：

```text
CTA 0 / warp 0: C[M [0,16),  N [0,32)]
CTA 0 / warp 1: C[M [16,32), N [0,32)]
CTA 0 / warp 2: C[M [32,48), N [0,32)]
CTA 0 / warp 3: C[M [48,64), N [0,32)]
```

CTA 1 的 `M` 范围同理，但 `N` 范围变成 `[32,64)`。

### Thread shape

`threadsPerWarp = [4,8]` 表示一个 warp band 继续分成 `4 x 8` 个 thread 区域：

```text
M size = 16 / 4 = 4
N size = 32 / 8 = 4

thread shape = 4 x 4
```

对选中的 `CTA 0 / warp 0`：

```text
thread id = t
thread_m_group = floor(t / 8)
thread_n_group = t % 8

M start = warpMStart + thread_m_group * 4
N start = ctaNStart + thread_n_group * 4
```

例如：

```text
T0: C[M [0,4),  N [0,4)]
T1: C[M [0,4),  N [4,8)]
T8: C[M [4,8),  N [0,4)]
```

### Value shape

`sizePerThread = [4,4]` 表示一个 thread 持有 `4 x 4` 个 logical value。可视化面板里的每个 `value cell` 是其中一个 logical element：

```text
value shape = 1 x 1
```

对选中的 thread：

```text
value id = v
value_m = threadMStart + floor(v / 4)
value_n = threadNStart + (v % 4)
```

例如 thread 0 里：

```text
v0:  C[M 0, N 0]
v1:  C[M 0, N 1]
v4:  C[M 1, N 0]
v15: C[M 3, N 3]
```

## IR 里最重要的 before / after 变化

### 1. Dot parent layout 改了

Before：

```text
#blocked8 = #ttg.blocked<{..., CGALayout = [[1, 0]]}>

A/B dot operand parent = #blocked8
tt.dot result = tensor<64x64xf32, #blocked8>
```

After：

```text
#blocked = #ttg.blocked<{..., CGALayout = [[0, 1]]}>

A/B dot operand parent = #blocked
tt.dot result = tensor<64x64xf32, #blocked>
```

这说明 PlanCTA 把 dot 的 CTA ownership contract 从旧的 `M split` 改成了新的 `N split`。

### 2. Accumulator 周围的 layout mismatch 减少

Before：

```text
#blocked -> #blocked8 -> tt.dot -> #blocked
```

After：

```text
#blocked -> tt.dot -> #blocked
```

这说明 PlanCTA 不只是改 dot 本身，还通过 cast insertion / propagation 让 loop-carried accumulator 和 dot result 对齐到同一个 layout。

### 3. A load 可能变成 CTA-broadcast-like

After IR 里出现：

```text
#blocked4 = #ttg.blocked<{..., CGALayout = [[0, 0]]}>
```

原因是当前输出沿 `N` split，而 A operand 是 `A[M,K]`，不包含输出的 `N` 维。两个 CTA 虽然负责不同的输出 `N` 分片，但会复用同一块 A 数据，所以 A 的 load 可以呈现类似 CTA broadcast / replicated 的语义。

## PlanCTA 做了哪两件事

第一步：在 dot 周围建立新的 layout contract。

```text
processDot
  read M/N/K and numCTAs
  choose splitM/splitN
  create new CGALayout
  create new accumulator layout
  create new A/B dot operand layout
```

第二步：沿 producer / consumer 链传播。

```text
insertCasts
  insert casts around dot
  propagate layout through loop/load/store/broadcast/elementwise users and producers
```

所以这个 pass 不是“只改一行 `CGALayout`”，而是把一个 CTA ownership decision 落实到 dot 以及周围的数据流。

## Compiler contract

Input contract：

- module 上已经有 `ttg.num-ctas = 2`
- IR 里已经有 `tt.dot`
- dot operand/result 已经有 TTGIR layout encoding

Output contract：

- dot parent layout 明确表达 CTA ownership
- A/B dot operand parent 和 accumulator/result 使用一致的 CTA split contract
- producer/consumer 链尽量传播同一个 CTA ownership model

Invariant：

- tensor shape 不变：A 仍是 `64x32`，B 仍是 `32x64`，C 仍是 `64x64`
- element type 不变：A/B 是 `f16`，accumulator 是 `f32`
- 数学语义不变：还是同一个 matmul tile
- 改变的是 layout / encoding / CTA ownership / layout conversions

## 为什么重要

如果没有 PlanCTA：

- `num_ctas=2` 不会被系统性落实到 dot 的 accumulator layout 上
- dot 可能继续使用 before IR 里旧的临时 parent layout
- dot 前后会保留更多 `ttg.convert_layout`
- 后续 RemoveLayoutConversions、MMA/WGMMA lowering、store lowering 不知道应该依赖哪一种 CTA ownership

最重要的是，IR 里会缺少一个清楚的答案：

```text
这两个 CTA 到底沿 M 还是沿 N 分摊输出 tile？
```

当前例子的答案就是：

```text
两个 CTA 沿 output N 维切分。
CGALayout = [[0, 1]]
CTA 0 shape = 64 x 32
CTA 1 shape = 64 x 32
```
