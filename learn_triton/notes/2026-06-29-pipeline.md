# 2026-06-29 学习笔记：Pipeline

## A. 定位（What & Where）

### 1. 一句话 + 在 pipeline 中的位置

本轮学习 `TritonGPUPipeline`。它位于软件流水主线的最后一段：

```text
... -> AssignLatencies
    -> ScheduleLoops
    -> Pipeline
    -> ...
```

一句话：

```text
Pipeline 不是再写一层 hint。
它真正消费 loop.stage / loop.cluster / tt.scheduled_max_stage，
先把 loop body 改写成 async copy / wait / barrier / TMEM token 这类资源流，
再把 schedule 展开成可执行的 prologue / kernel / epilogue。
```

这里反复提到的 `hot loop`，就是 kernel 里最耗时、最值得优化的那层主循环。
在 matmul 语境里，通常就是沿 `K` 维反复做 load / dot / pointer update 的那层 loop。

先给一个最重要的预警：

```text
这批 canonical matmul 样本不适合拿来学习典型的多 stage overlap。
更完整的根因和三架构证据见 §11.4；
真正展示多 stage 的主样本应看 §12 的 lit 测试。
```

- 这份 before 的 hot loop 明确带了：
  `loop.stage = 0` / `loop.cluster = 0`，并且 loop 本身有
  `tt.scheduled_max_stage = 0`，见 [084_Before](/LocalRun/jiangzhe.zhao/my_repo/triton/learn_triton/dumps/matmul/sm100_num_ctas1/mlir-pass-dump.split/084_Before_TritonGPUPipeline.mlir:83)
- 本地 `mark_effective_passes.py` 结果里，`AssignLatencies` / `ScheduleLoops` 都是 `CHANGED`

#### 4.4 结论

```text
三个架构在 Before 阶段已经分叉了，而且分叉有两层：

第一层：更早的 arch-specific matmul lowering 已经不同
  sm86: tt.dot / mma.sync world
  sm90: WGMMA memdesc world
  sm100: MMAv5 + TMEM world

第二层：Pipeline contract 的“成熟度”也不同
  sm86: 没看到可序列化 schedule contract
  sm90: 没看到可序列化 schedule contract
  sm100: 输入有 schedule contract，但 max stage = 0，只是单 stage contract
```

所以这三份 canonical dump 的 after 差异，不能直接归因为 `Pipeline` 自己“做了三套完全不同的事”；更准确的说法是：

```text
它继承了 before 已分叉的输入，
并且共同受到一个更上游的根因约束：
这几个 canonical 样本没有形成真正可展开的多 stage contract。
```

## B. 机制（How it works）

### 5. Compiler Decision

`Pipeline` 回答的问题不是“哪些 op 该在 stage 0/1/2”，这个已经由
`ScheduleLoops` 回答了。它现在回答的是：

```text
给定已有 coarse schedule，
编译器应该怎样把这个 schedule 物化成真实可执行的 loop 结构和异步资源流？
```

更细一点，分成两个子问题：

```text
Lowering decision:
  哪些 load 应该变成 async global->local / TMA async copy？
  哪些 MMA/TMEM 需要 barrier、wait、额外 buffer、multibuffer？

Expansion decision:
  现有 stage schedule 应该怎样展开成 prologue / steady-state / epilogue？
  哪些跨 stage value 需要额外 loop-carried versions？
```

#### 5.1 Decision Tree

```text
PipelinePass::runOnOperation
  -> lowerLoops(moduleOp)
       -> lowerMMAs(forOp, schedule)
       -> lowerLoads(forOp, schedule, axisInfoAnalysis)
       -> lowerTMADescriptors(forOp, schedule)
  -> expandLoops(moduleOp)
  -> removePipeliningAttributes(moduleOp)
  -> pipelineWgmma(moduleOp, numStages)
  -> updateWaits(moduleOp)
  -> arith canonicalization
  -> if numStages > 1:
       pipelineTMAStores(forOp)
```

### 6. Compiler Contract

#### 6.1 Input contract

- `scf.for` 已经有可反序列化的 coarse schedule，来自 `loop.stage` /
  `loop.cluster` / `tt.scheduled_max_stage`，见 [LowerLoops.cpp:1048](/LocalRun/jiangzhe.zhao/my_repo/triton/lib/Dialect/TritonGPU/Transforms/Pipeliner/LowerLoops.cpp:1048)
- loop 只支持 distance-1 loop-carried dependency，且所有 scheduled op 都在
  top-level loop body block，见 [PipelineExpander.cpp:184](/LocalRun/jiangzhe.zhao/my_repo/triton/lib/Dialect/TritonGPU/Transforms/Pipeliner/PipelineExpander.cpp:184)
- 对真正要展开的 loop，schedule 不能为空且必须自洽，见 [PipelineExpander.cpp:138](/LocalRun/jiangzhe.zhao/my_repo/triton/lib/Dialect/TritonGPU/Transforms/Pipeliner/PipelineExpander.cpp:138)

#### 6.2 Output contract

- 需要异步化的 load 不再只是普通 `tt.load`，而是会被改成：
  `ttg.async_copy_global_to_local` / `ttng.async_tma_*` +
  `ttg.async_commit_group` + `ttg.async_wait` + `ttg.local_load`
- 需要流水化的 MMAv5 path 会显式得到 barrier / wait / TMEM token /
  多 buffer state
- loop 会被展开为真实的 software pipeline 结构，而不只是 metadata
- 展开后原始 `loop.stage` / `loop.cluster` 等中间 metadata 会被清理，见 [SoftwarePipeliner.cpp:195](/LocalRun/jiangzhe.zhao/my_repo/triton/lib/Dialect/TritonGPU/Transforms/Pipeliner/SoftwarePipeliner.cpp:195)

#### 6.3 Next pass relies on

- 更后面的 lowering 可以直接依赖已经显式化的 async copy / barrier / wait / token
- 不再需要从 `loop.stage` 倒推 pipeline 结构，因为 `Pipeline` 已经把结构展开了

#### 6.4 Invariant

`Pipeline` 改得很大，但这些语义不该变：

- logical tensor shape 不变
- 数学结果不变
- loop iteration 的逻辑语义不变
- memory 地址的逻辑含义不变

改变的是：

- 执行顺序的显式化
- buffer 生命周期
- async token / barrier / wait 的显式同步
- loop 结构从“单个原始 `scf.for`”变成“prologue + steady-state + epilogue”

### 7. 两层模型：`lowerLoops` vs `PipelineExpander`

#### 7.1 第一层：`lowerLoops` 改 op，不改整体 loop 形状

这一层的职责不是展开 `scf.for`，而是把 loop body 的关键 op 改成“可流水化物件”。

最典型是普通 global load：

- 原始 `tt.load`
- 变成 `ttg.async_copy_global_to_local`
- 再接 `ttg.async_commit_group`
- 在 consumer stage 插 `ttg.async_wait`
- 最后用 `ttg.local_load` 取回数据

证据直接看 `createAsyncCopy`：

- 创建 async copy 见 [LowerLoops.cpp:169](/LocalRun/jiangzhe.zhao/my_repo/triton/lib/Dialect/TritonGPU/Transforms/Pipeliner/LowerLoops.cpp:169)
- 创建 commit 见 [LowerLoops.cpp:174](/LocalRun/jiangzhe.zhao/my_repo/triton/lib/Dialect/TritonGPU/Transforms/Pipeliner/LowerLoops.cpp:174)
- 创建 wait 和 local_load 见 [LowerLoops.cpp:177](/LocalRun/jiangzhe.zhao/my_repo/triton/lib/Dialect/TritonGPU/Transforms/Pipeliner/LowerLoops.cpp:177)

#### 7.2 第二层：`PipelineExpander` 展开 loop 结构

这一层才是真正的 software pipelining expander。

它会：

- 读取 final schedule
- 验证 schedule 合法性
- 构建新的 kernel loop
- 复制/重映射跨 stage values
- 生成 prologue / kernel / epilogue

关键入口：

- `initializeLoopInfo` 读取 schedule、检查 dynamic loop 和 legality，见 [PipelineExpander.cpp:130](/LocalRun/jiangzhe.zhao/my_repo/triton/lib/Dialect/TritonGPU/Transforms/Pipeliner/PipelineExpander.cpp:130)
- `createKernelLoop` 会把 `ub` 调整为 `ub - maxStage * step`，为 epilogue 留出 peeled iterations，见 [PipelineExpander.cpp:455](/LocalRun/jiangzhe.zhao/my_repo/triton/lib/Dialect/TritonGPU/Transforms/Pipeliner/PipelineExpander.cpp:455)
- `createKernel` 按 stage clone op，并把跨 stage operand remap 到额外 loop-carried versions，见 [PipelineExpander.cpp:480](/LocalRun/jiangzhe.zhao/my_repo/triton/lib/Dialect/TritonGPU/Transforms/Pipeliner/PipelineExpander.cpp:480)
- `emitEpilogue` 生成尾部阶段，见 [PipelineExpander.cpp:657](/LocalRun/jiangzhe.zhao/my_repo/triton/lib/Dialect/TritonGPU/Transforms/Pipeliner/PipelineExpander.cpp:657)

所以要把两个层次分开记：

```text
LowerLoops:
  Triton-specific resource lowering

PipelineExpander:
  generic stage-scheduled loop expansion
```

### 9. 术语表 / 关键概念澄清

#### 9.1 `distance-1 loop-carried dependency`

```text
第 i 次迭代产生的 loop-carried state，
只能直接传给第 i+1 次迭代使用，
不能要求 expander 直接跟踪 i -> i+2 / i+3 这种更远依赖。
```

在 `scf.for` 语义里，最典型的形状就是 `iter_args` / `scf.yield`：

```text
当前轮 yield 出去的值，
作为下一轮 region iter arg 进来。
```

#### 9.2 `top-level loop body block`

这里的 `top-level loop body block`，就是 `scf.for` 的直接 body block，也就是
`forOp.getBody()` 本身。

例如：

```text
scf.for {
  %a = ...
  scf.if %cond {
    %x = ...
  }
  %b = ...
}
```

- `%a` / `%b` 是 top-level loop body block 里的 op
- `scf.if` 这个 op 本身也是 top-level op
- 但 `if` 内部 region 里的 `%x` 不是 top-level op

当前 expander 只支持“单 block、线性 program order”的 schedule，不直接 pipeline 嵌套控制流里的 op。

#### 9.3 `schedule` 自洽

这里的“自洽”不是抽象的“看起来合理”，而是：

```text
stage 分配后的整体执行顺序，不能违反真实 def-use 依赖。
consumer 不能被排到 producer 前面。
```

`verifySchedule()` 会先把每个 op 映射到一个展开后的线性 cycle：

```text
unrolledCycle = program-order-in-block + stage * numOpsPerIter
```

然后检查所有 operand 的 producer 是否仍然在 consumer 之前；若是 loop-carried dependency，
再按 dependency distance 做修正。

#### 9.4 `ttg.async_wait`

`ttg.async_wait {num = N}` 的语义要记精确：

```text
不是“等 N 个 group”。
而是“等到 outstanding async copy groups 的数量 <= N”。
num = 0 才是等到全部完成。
```

定义直接写在 [TritonGPUOps.td:47](/LocalRun/jiangzhe.zhao/my_repo/triton/include/triton/Dialect/TritonGPU/IR/TritonGPUOps.td:47)。

它不是“查询当前还剩多少组没完成”，而是一个显式的执行边界：

```text
程序必须至少等到 outstanding async copy groups 的数量 <= num，
才能继续往下执行。
```

它的意义有三层：

- 保证 consumer 读 shared/local buffer 前，对应那批 copy 已经 ready
- 允许“部分等待”而不是每次都全等完，从而保留 pipeline overlap
- 给编译器一个可插入、可移动、可合并、可重算 `num` 的真实 IR 节点

它和 `ttg.barrier` 的区别见 §10.3。

#### 9.5 怎么保证“等到的是我要的那批数据”

不是靠 `async_wait` 自己去“点名”某个 tile；而是靠三件事一起保证：

- `async_commit_group` 形成有先后顺序的 group
- multibuffer / ring-buffer index 决定“第几轮读哪个 buffer slot”
- `async_wait {num = N}` 只负责保证“更老的那些 group 已经完成到可消费程度”

所以 consumer 不是“随便拿一份已经来了的数据”，而是：

```text
按静态决定好的 slot 去读；
wait 只保证这个老 slot 对应的老 group 已经按顺序 ready。
```

#### 9.6 `local`

`ttg.local_load` / `ttg.async_copy_global_to_local` 里的 `local` 不该按 CUDA 术语里的 per-thread local memory 去理解。
在 Triton 这条 pipeline 主路径里，它基本就是：

```text
CTA-local 的 staging buffer，
通常由 SharedMemory-backed 的 MemDesc 表示。
```

### 10. 为什么要这样改 + GPU / Hardware reason

#### 10.1 仅有 `loop.stage` 还不够

`ScheduleLoops` 产出的只是 coarse contract：

```text
这个 op 在 stage 0
那个 op 在 stage 2
```

但硬件不会自动帮你做这些事情：

- 提前发 global->shared copy
- 保留多份 buffer
- 在 consumer 前插 wait
- 在 loop 头尾补 prologue / epilogue

这些都必须 `Pipeline` 显式改 IR。

#### 10.2 GPU / hardware reason

```text
Memory reason:
  用 async cp / TMA 提前把数据搬到 shared/local buffer。

Execution reason:
  让 iteration i 的 compute 和 iteration i+1 的 load overlap。

Resource reason:
  shared memory / TMEM / barrier 都需要显式分配和轮换。

Instruction reason:
  WGMMA / MMAv5 的 async issue 和 wait 需要显式 token/barrier contract。
```

#### 10.3 为什么“等到数据到了”仍然还需要 barrier

`ttg.async_wait` 和 `ttg.barrier` 解决的是两个不同问题：

- `async_wait`：copy completion，保证数据搬运已经完成到可消费程度
- `ttg.barrier`：CTA 内线程 rendezvous，保证相关线程都已经走到这个共同点

这里的 `rendezvous` 就是“会合点”的意思：CTA 里的线程要先到齐，再一起继续。

在 cooperative tiled pipeline 里，常见形状是：

- 每个线程只负责搬 tile 的一部分到 shared buffer
- 但所有线程都要消费整个 tile

这时即使某些 async copy 已经完成，也不等于：

- 生产这个 tile 的所有线程都已经到齐
- 整个 tile 作为一个整体已经可以被全 CTA 一致消费

所以正确的语义是：

```text
async_wait 管“货到了没有”
ttg.barrier 管“人到齐没有”
```

两者并不天然等价。

## C. 证据（What we actually see）

### 11. 三架构证据合一：sm86 / sm90 / sm100

#### 11.1 `sm86`

- 对 hot matmul loop 来说，几乎可以视为 body-level no-op
- `Before/After` 正文基本不变；`mark_effective_passes.py` 报出来的 `CHANGED` 主要被内部 debug dump 污染
- 核心原因是：hot loop 上没有 serialized schedule contract，上游 `AssignLatencies` / `ScheduleLoops` 也是 `no-op`

#### 11.2 `sm90`

- pass 是 effective 的，但主要体现在 WGMMA 专项后处理
- 不是“load pipeline 成功展开”，而是 “WGMMA async/wait contract 被显式化”
- `sm90` 看起来像例外，因为它明明没有多 stage schedule，却还是插入了：
  - `ttng.warp_group_dot {isAsync = true}`
  - `ttng.warp_group_dot_wait`
- 这不是 load 流水起来了，而是 driver 后半段的 WGMMA 专项后处理单独生效

#### 11.3 `sm100`

- pass 明显 effective
- 更准确地说，输入是 `tt.scheduled_max_stage = 0` 的单 stage contract
- 但 pass 内部的 `LowerLoops/ExpandLoops` 已经把它物化成 async MMAv5 pipeline 形态
- 它仍然不是学习典型 multi-stage load overlap 的最佳样本

#### 11.4 三者的共同根因

```text
这三个 canonical matmul 样本里，Pipeline 的表现差异，大体来自同一个根因：
hot loop 没有形成“可展开的多 stage schedule contract”。
```

更精确地说：

- `sm86`：没有看到 serialized schedule attr；上游 `AssignLatencies` / `ScheduleLoops` 也是 `no-op`
- `sm90`：同样没有看到 serialized schedule attr；上游 `AssignLatencies` / `ScheduleLoops` 也是 `no-op`
- `sm100`：输入有 serialized schedule，但 `tt.scheduled_max_stage = 0`，只是单 stage contract；
  pass 内部物化后会出现 async MMAv5 pipeline 形态，但这不是典型的多 stage load overlap 样本

所以这里不是：

```text
Pipeline 在 Ampere 不会做事，在 Hopper 只会做一点，在 Blackwell 才会做很多事。
```

更准确的是：

```text
Pipeline 只有在上游已经交付“可展开 contract”时，才会展开多 stage overlap。
这批 canonical 样本里，那个 contract 基本不存在，或者只存在单 stage 版本。
```

#### 11.5 一个方法论上的补充

我本地跑了：

```text
python learn_triton/tools/mark_effective_passes.py learn_triton/dumps/matmul/sm86_num_ctas1/mlir-pass-dump.log
python learn_triton/tools/mark_effective_passes.py learn_triton/dumps/matmul/sm90_num_ctas1/mlir-pass-dump.log
python learn_triton/tools/mark_effective_passes.py learn_triton/dumps/matmul/sm100_num_ctas1/mlir-pass-dump.log
```

结果里三者的 `TritonGPUPipeline` 都会显示成 `CHANGED`。但这里要小心：

```text
这个 pass 开了 dump-intermediate-steps=true。
当前 split dump 的 Before 文件里混入了
"SoftwarePipeliner internal IR Dump After: LowerLoops/ExpandLoops" 这类内部子 dump。

所以对 Pipeline 这个 pass，
mark_effective_passes.py 的 [CHANGED] 可以当“值得看”的提示，
但不能直接当“hot loop body 一定发生了语义性变化”的硬证据。
```

真正的硬证据还是要看 `Before/After` 正文 diff：

- `sm86` 的正文几乎不变，只有 dump 头和内部调试块差异
- `sm90` 的正文变化集中在 `warp_group_dot` async/wait
- `sm100` 的正文变化是 barrier/TMEM/async MMA 资源流落地

#### 11.6 Architecture Evolution

##### Ampere

```text
Hardware changed:
  仍是 mma.sync / cp.async 这代基线。

Compiler changed:
  需要表达 shared-memory async copy 和普通 tensor-core path。

Pass changed:
  在真正有多 stage schedule 时，Pipeline 主要负责 async cp + ring buffer + expander。
  但当前 canonical 样本没有拿到这个 contract，所以这里只看到近似 no-op。
```

##### Hopper

```text
Hardware changed:
  引入 WGMMA / warp-group 执行模型。

Compiler changed:
  IR 必须表达 warp-group shared-mem operand 和 async dot wait contract。

Pass changed:
  即使没有多 stage load pipeline，Pipeline 仍可能通过 WGMMA 后处理
  把 warp_group_dot 变成 async launch + wait。
```

##### Blackwell

```text
Hardware changed:
  MMAv5、TMEM、barrier 资源流更重要。

Compiler changed:
  IR 必须显式表达 barrier init/wait/invalidate、TMEM token、async MMA。

Pass changed:
  Pipeline 在 Blackwell 上不只是搬 load，更像“把 MMAv5 执行资源图显式化”。
  当前 canonical 输入仍然只有单 stage contract，但 pass 内部已能物化出 async MMAv5 pipeline 形态。
```

## D. 判断与延伸（Judgment & next）

### 13. 谁决定 pipeline 优不优 + Alternative Design

先给结论：

```text
没有任何一个 pass 在保证“最优”。
真正主导 pipeline 形状的是 AssignLatencies + ScheduleLoops；
Pipeline 负责把这个形状物化，而不是重新做全局优化搜索。
```

为什么这么说：

- `AssignLatencies` 决定哪些 op 是 latency anchor，以及 latency 值大致是多少
- `ScheduleLoops` 用 latency-weighted longest-path heuristic 决定 `loop.stage` / `loop.cluster`
- `Pipeline` 只消费这些 attrs，做 lowering 和 expansion

所以如果问“哪个 pass 最决定 pipeline 看起来优不优”，答案是：

```text
主要是 ScheduleLoops，
但它也是启发式决定，不是最优求解器。
```

它主要优化的是 latency hiding，而不是完整硬件代价模型。当前实现没有同时全局建模：

- register pressure
- shared-memory / TMEM pressure
- occupancy
- issue width
- barrier / wait overhead
- prologue / epilogue 额外成本

因此更准确的表述是：

```text
Triton 当前做的是“合法、通常有收益、偏 latency-hiding 的 pipeline 构造”，
不是带最优性证明的 modulo scheduling / global search。
```

#### 13.1 Alternative Design

##### 不做 multibuffer

```text
Alternative:
  保持同步 load，不引入 alloc / insertIdx / extractIdx。

Cost:
  load 和 compute 不会 overlap，ScheduleLoops 产出的 stage 信息基本浪费。
```

##### 上游给不同的 stage 数

```text
Alternative:
  上游只给 1 stage，或者给 2/3/4 stage。

Cost:
  stage 太少，隐藏不了 latency；
  stage 太多，会拉长活跃区间、增加 shared/TMEM/barrier 压力。
```

这也是为什么：

```text
Pipeline 自己不决定“要不要多 stage”。
它消费上游 contract，而不是在这里重新发明 schedule。
```

##### Hopper/Blackwell 路径不用专门后处理

```text
Alternative:
  把 WGMMA / TMA / MMAv5 都塞进统一的 generic expander 逻辑。

Cost:
  generic expander 会变得过度了解架构细节；
  当前设计把架构专属资源逻辑保留在 lowerLoops / pipelineWgmma 这层，边界更清楚。
```

##### 不 peel epilogue

```text
Alternative:
  不生成 peeled epilogue，完全依赖更复杂的 loop 内谓词化。

Cost:
  steady-state kernel 会更脏，expander 合法性和代码质量都更难维护。
```

##### `Pipeline` 主要是不是“插同步 + 展开 loop”

可以这样概括，但要稍微说严一点：

```text
第一步：
  把 load / MMA / TMEM / barrier 相关 op 改写成显式 async resource flow
  （async copy / commit / wait / barrier / token / multibuffer state）

第二步：
  按已有 stage schedule 把 loop 展开成真正的
  prologue / steady-state / epilogue
```

所以“插同步”只是第一步里的一部分。更完整的说法是：

```text
Pipeline 先把 schedule 物化成 async/resource-flow IR，
再把这个 schedule 展开成真实执行结构。
```

### 14. 没有这个 pass 会怎样 + Open Questions + Knowledge Card + 后续

#### 14.1 没有这个 pass 会怎样

##### Correctness / lowering

- 后续不会自动把 coarse schedule 变成真实 loop 结构
- MMAv5 / WGMMA 的 async contract 不会显式落地
- barrier / wait / token / multibuffer state 缺失

##### Performance

- load 与 compute 无法实际 overlap
- shared/TMEM 多缓冲无法建立
- 即便前面有 `loop.stage`，也只是 metadata，不会带来真实 latency hiding

##### Compiler pipeline

- `ScheduleLoops` 的输出 contract 没有 consumer
- 之前写下的 `loop.stage` / `loop.cluster` 只会停留在纸面上

#### 14.2 Open Questions

- 为什么这组 canonical `sm86` / `sm90` matmul 在当前配置下没有形成 serialized multi-stage contract？
  是 kernel 参数、`num_stages` 选择，还是上游 schedule legality 限制导致的？
- `sm100` canonical matmul 为什么只得到 `tt.scheduled_max_stage = 0`？
  是当前 kernel 结构不值得 overlap，还是 Blackwell 这条主线对 canonical matmul 还没激励到多 stage？
- `pipelineTMAStores` 的收益在 Blackwell canonical matmul 上什么时候会真正显现？
  关键前提显然是 `numStages > 1`，但具体还受哪些 legality 条件限制？

#### 14.3 Knowledge Card

```text
Pass:
  TritonGPUPipeline

Compiler question:
  已有 coarse schedule 要怎样变成真实可执行的 async resource flow + pipelined loop？

Input contract:
  loop.stage / loop.cluster / tt.scheduled_max_stage 已经存在并且合法。

Output contract:
  async copy / wait / barrier / token / prologue / kernel / epilogue 显式落地。

Two-layer model:
  lowerLoops = Triton-specific resource lowering
  expandLoops = generic stage-scheduled loop expansion

Main caveat:
  没有多 stage contract 时，Pipeline 不会凭空制造 overlap。
```

#### 14.4 后续建议

学完 `Pipeline` 之后，最自然的下一步有两个：

- `WarpSpecialize`
  你会看到 schedule 不再只是单一 loop pipeline，而是开始和 partition / warp role 绑定。

- `AllocateSharedMemory`
  你会看到 `Pipeline` 引入的 local/shared/TMEM resource，后面怎样真正分配物理 offset。
